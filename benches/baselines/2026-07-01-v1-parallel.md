# Bounded parallel v1 hashing — 2026-07-01

This report compares the retained sequential correctness oracle (`--threads 1`)
with the new bounded two-worker pipeline (`--threads 2`). The automatic policy uses
two workers when at least two logical CPUs are available and falls back to the
sequential path on a single CPU. Explicit counts remain per operation; no global
thread pool is configured.

## Design

- One reader preserves the v1 logical concatenated file stream and snapshot checks.
- Pieces carry monotonically increasing sequence numbers.
- Scoped worker threads hash independent complete pieces.
- Ordered collection restores sequence and emits monotonic progress.
- A reusable buffer pool and bounded job channel cap payload buffers at
  `(workers + queue capacity) × piece length`, plus a 64 KiB read buffer.
- Cancellation or worker failure closes the operation and joins scoped threads.
- `HashThreads::Exact(1)` remains the explicit sequential oracle.

## Correctness

Differential tests cover generated multifile streams, pieces crossing file
boundaries, one-slot backpressure, ordered progress, forced out-of-order worker
completion, injected worker failure, preflight and mid-flight cancellation, option
bounds, CLI parity, and Python parity. Every measured torrent was byte-identical
between one and two workers and passed both `btpc validate` and `btpc verify`.

## Kernel comparison

Criterion command:

```bash
cargo bench -p btpc-core --bench core -- v1_piece_hashing \
  --warm-up-time 0.1 --measurement-time 0.3 --sample-size 15 --noplot
```

| Path | Estimate | Throughput | Relative |
|---|---:|---:|---:|
| sequential | 3.8835 ms | 2.0117 GiB/s | 1.00× |
| parallel, 4 workers | 1.1553 ms | 6.7624 GiB/s | 3.36× |

The four-worker microbenchmark proves hashing parallelism is material. End-to-end
measurements below select two workers because four added memory and variability
without a stable wall-time improvement on this machine.

## End-to-end comparison

Environment matches `2026-07-01-pre-optimization.md`: Apple M3 Max, 14 cores,
38.65 GB RAM, macOS 26.5, APFS, Rust 1.85.0. Both deterministic inputs contain
128 MiB and use 4 MiB pieces. The large file seed is `3001`; the 256-file mixed
tree seed is `3002`. Ten randomized blocked rounds compared one versus two workers.
Raw ignored artifacts are under `benchmark-results/todo-30/final/`.

| Dataset | Threads | Median | Mean | SD | MAD | CV | Median MiB/s | Max RSS |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| one 128 MiB file | 1 | 101.646 ms | 101.973 ms | 2.665 ms | 1.509 ms | 2.61% | 1,259.28 | 11,419,648 B |
| one 128 MiB file | 2 | 74.242 ms | 111.675 ms | 80.401 ms | 3.370 ms | 71.99% | 1,724.08 | 24,100,864 B |
| 256 × 512 KiB tree | 1 | 129.363 ms | 153.884 ms | 59.148 ms | 0.971 ms | 38.44% | 989.46 | 12,222,464 B |
| 256 × 512 KiB tree | 2 | 74.995 ms | 84.484 ms | 22.300 ms | 0.928 ms | 26.40% | 1,706.78 | 24,838,144 B |

Median speedup is **1.37×** for the large file and **1.72×** for the mixed tree.
Peak RSS increases by roughly 12–13 MiB, consistent with four total 4 MiB piece
buffers plus allocator/process overhead. The two-worker large-file mean/CV include
one scheduler outlier; medians and MAD remain substantially better. These are local
diagnostic results, not release or competitor claims.
